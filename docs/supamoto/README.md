# SupaMoto USSD Customization Documentation

This directory contains comprehensive documentation of all customizations, features, and modifications made to the original `ixoworld/ixo-ussd` repository specifically for the **SupaMoto customer**.

## Overview

SupaMoto is a specialized implementation of the IXO USSD platform designed to support bean distribution workflows, customer activation, and role-based access control for lead generators and agents. This fork extends the vanilla IXO USSD with customer-specific features while maintaining compatibility with the core architecture.

## Key SupaMoto Features

### 1. **Role-Based Access Control (RBAC)**

- **Customer Roles**: `customer`, `lead_generator`, `call_center`, `admin`
- **Database Support**: Role column added to `customers` table with CHECK constraint
- **Menu Differentiation**: Different menu options displayed based on user role
- **Access Control**: Lead generators can access Agent Tools; customers access Customer Tools

### 2. **Bean Distribution Workflow**

- **Lead Generator Flow**: Activate customers, register intent to deliver beans, verify OTP, confirm delivery
- **Customer Flow**: Receive activation SMS, activate account, claim 1,000 Day Household eligibility, confirm bean receipt
- **OTP System**: 6-digit OTPs valid for 10 minutes (configurable) for bean distribution verification
- **Dual Confirmation**: Both LG and customer must confirm bean delivery within 7-day window

### 3. **Customer Activation System**

- **Temporary PINs**: Generated by lead generators during customer verification at distribution points
- **Activation SMS**: Sent to customers with temporary PIN and instructions
- **PIN Consolidation**: Temporary and permanent PINs stored in `customers.encrypted_pin`
- **Audit Trail**: PIN resets tracked in `audit_log` table for security

### 4. **1,000 Day Household Claims**

- **Self-Proclamation**: Customers declare if they have eligible 1,000 Day Household
- **Eligibility Criteria**: Family with pregnant/breastfeeding mother or child under 2 years old
- **Bean Voucher Allocation**: Approved claims receive bean vouchers for distribution
- **Claim Tracking**: Stored in `household_claims` table with status and bot response

### 5. **SMS Integration**

- **Template System**: Centralized SMS templates in `src/templates/sms/`
- **Activation SMS**: Sent after LG activates customer
- **OTP SMS**: Sent to customer for bean distribution verification
- **LG Notifications**: SMS sent to lead generators with voucher status and OTP instructions
- **Error Handling**: SMS failures tracked in audit log

### 6. **Enhanced Menu Structure**

- **Pre-Menu**: Know More, Account Menu, Services (if authenticated), Exit
- **Account Menu**: Login, Create Account
- **Services Menu**: Customer Tools or Agent Tools (based on role)
- **Customer Tools**: 1,000 Day Household claim, Confirm Bean Receipt
- **Agent Tools**: Activate Customer, Register Intent, Submit OTP, Confirm Delivery

## Database Schema

The database schema is initialized via a single consolidated migration file (`migrations/postgres/000-init-all.sql`) that creates all tables, indexes, and constraints in one idempotent script.

**Core Tables**:

- `phones`, `households`, `customers`, `wallets` - Base customer and identity data
- `customers.role` - Role-based access control (customer, lead_generator, call_center, admin)
- `customers.encrypted_pin` - Consolidated PIN storage (temporary and permanent)

**Bean Distribution Tables**:

- `lg_delivery_intents` - Lead generator intent to deliver beans
- `bean_distribution_otps` - OTP tracking (10-minute validity)
- `bean_delivery_confirmations` - Dual confirmations (LG + Customer) within 7-day window

**Survey & Claims Tables**:

- `household_claims` - 1,000 Day Household claims with embedded survey responses
- `household_claims.survey_form` - JSONB field storing complete survey form and answers

**Audit Tables**:

- `audit_log` - Security events, failed SMS, denied receipts, account locks

## State Machine Architecture

### Parent Machine (`supamotoMachine`)

- **Orchestrator**: Main entry point for all USSD flows
- **Child Machines**: Delegates to specialized machines for different flows
- **States**: idle, preMenu, knowMore, accountMenu, login, accountCreation, userMainMenu, error, closeSession

### User Services Machine (`userServicesMachine`)

- **Role-Based Routing**: Routes to Customer Tools or Agent Tools based on role
- **Customer Tools**: 1,000 Day Household claims and bean receipt confirmation
- **Agent Tools**: Customer activation, intent registration, OTP submission, delivery confirmation

### Customer Activation Machine (`customerActivationMachine`)

- **LG Flow**: Verify customer, enter phone, send activation SMS, wait for customer
- **Customer Flow**: Receive activation SMS, enter temporary PIN, activate account

### Account Menu Machine (`accountMenuMachine`)

- **Login**: Authenticate existing customers with PIN
- **Account Creation**: Register new customers

## Configuration & Environment

### Key Environment Variables

- `OTP_VALIDITY_MINUTES`: OTP expiration time (default: 10 minutes)
- `SERVICE_CODES`: USSD service codes (e.g., `*2233#`)
- `SMS_PROVIDER`: SMS delivery provider configuration
- `DATABASE_URL`: PostgreSQL connection string

### Branding & Messages

- Centralized in `src/constants/branding.ts`
- Customizable welcome, goodbye, and menu messages
- Supports multiple languages via `preferred_language` field

## Security Features

### PIN Management

- **Encryption**: All PINs encrypted at rest using bcrypt
- **Session PINs**: Ephemeral PINs used for Matrix vault decryption in-session
- **PIN Reset Audit**: All PIN resets logged with timestamp and context
- **Temporary PINs**: 6-digit codes with expiration for customer activation

### Session Management

- **Session IDs**: Unique identifiers for each USSD session
- **Session Timeout**: Configurable session expiration
- **Phone Number Tracking**: All interactions tracked by phone number
- **Audit Logging**: Comprehensive audit trail for security events

## Integration Points

### External Services

- **IXO Blockchain**: Web3 integration for wallet creation
- **Claims Bot**: API service for submitting claims via `@ixo/supamoto-bot-sdk`
  - Lead creation claims (submitted during customer account creation)
  - 1,000 Day Household survey claims
  - **SDK Documentation**: See [SUPAMOTO_BOT_SDK_GUIDE.md](./SUPAMOTO_BOT_SDK_GUIDE.md) for comprehensive SDK usage guide
- **Matrix**: Secure vault for encrypted data storage
- **SMS Provider**: Africa's Talking or similar for SMS delivery
- **Subscriptions Service**: Bean voucher verification and allocation

### Data Flow

- Phone → Customer → Wallet (IXO Profile + Account) → Matrix Vault
- Customer activation triggers IXO profile creation
- Lead creation claim submitted via Claims Bot SDK (HTTP API)
- Bean distribution verified against subscription service

### Claims Submission Architecture

**Current Approach** (as of migration):

- **Lead Creation Claims**: Submitted via `@ixo/supamoto-bot-sdk` HTTP API to Claims Bot service
- **1,000 Day Household Claims**: Submitted via `@ixo/supamoto-bot-sdk` HTTP API to Claims Bot service

**Benefits**:

- Simplified implementation (no blockchain signing complexity)
- Faster response times (HTTP API vs blockchain transaction)
- No gas fees or feegranter configuration needed
- Centralized claim tracking in Claims Bot service

## Database Initialization

### Schema Deployment

The database schema is deployed using a single consolidated initialization script:

**File**: `migrations/postgres/000-init-all.sql`

**Contents**:

1. Core Tables (Phones, Households, Customers, Wallets)
2. Customer Activation & Eligibility
3. Bean Distribution & Delivery
4. Household Survey & Claims (with JSONB storage)
5. Audit & Logging
6. Performance Indexes

**Deployment**: Run `pnpm build && node dist/migrations/run-migrations.js`

**Note**: The script is idempotent and can be safely re-run. It drops and recreates all tables, making it suitable for fresh deployments and development environments.
